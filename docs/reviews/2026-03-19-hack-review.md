# Stringification And Heuristic-Hack Review

Date: 2026-03-19

Scope:

- Current repository state
- Misuse of display-oriented APIs such as `Pretty`, `String`, and `PrettyValue`
- Places where names, labels, or textual shape carry semantic meaning
- Distinguishing acceptable sugar encodings from brittle implementation hacks
- Includes both confirmed problematic uses and structurally risky heuristics

This review focuses on a narrow but important failure mode:

- comparing or keying by pretty-printed text
- treating stringified output as semantic identity
- inferring semantics from internal naming conventions rather than explicit metadata
- using partial textual encodings as stand-ins for structural data

## Executive Judgment

The good news is that the repository does not appear to be broadly abusing
`types.Pretty` or `Value.String()` for direct equality checks.
I did not find implementation code that does `Pretty(a) == Pretty(b)` or uses
`PrettyValue(...)` as a semantic branch condition.

However, there are still several brittle areas.

The most important confirmed issue is:

- type-family reduction cache keys still fall back to `types.Pretty(...)` for unsupported type nodes

Beyond that, the main pattern is not "pretty-print equality" but
"name/shape heuristics standing in for real metadata":

- `$` in a name means "compiler-generated"
- `$Dict` suffix means "dictionary type"
- instance dictionary identity is encoded by a lossy string name
- tuples are detected by record labels `_1`, `_2`, ...

Some of these are acceptable surface encodings.
Some are real maintenance risks.
The review below separates those two classes.

## Findings

## 1. High: type-family reduction cache keys still fall back to `types.Pretty`

Severity: High

Status:

- unresolved
- confirmed in current implementation
- directly matches the "display API used for semantic identity" concern

Relevant code:

- [reduce.go](/Users/cwd-k2/Projects/gicel/internal/check/family/reduce.go#L266)
- [reduce.go](/Users/cwd-k2/Projects/gicel/internal/check/family/reduce.go#L277)
- [reduce.go](/Users/cwd-k2/Projects/gicel/internal/check/family/reduce.go#L327)
- [type.go](/Users/cwd-k2/Projects/gicel/internal/types/type.go#L97)

Problem:

`familyAppKey` is intended to build a structural cache key for type-family applications.
Most common type nodes are handled explicitly by `writeTypeKey`.
But its default branch falls back to:

- `fmt.Fprintf(b, "<%s>", types.Pretty(t))`

This means the cache key is no longer structural for any type node not covered by the switch.

Why it matters:

- cache identity should not depend on a human-oriented pretty-printer
- `Pretty` can change for formatting reasons without changing semantics
- different structures can, in principle, collapse to the same printed text
- the fallback is currently reachable because `writeTypeKey` does not handle at least `TyEvidence` and `TyError`

Assessment:

This is the clearest confirmed misuse found in the review.
It is not just "string-based infrastructure"; it is semantic cache identity
falling back to presentation logic.

Recommended direction:

- make `writeTypeKey` total over all `types.Type` variants
- if that is not possible, fail loudly rather than silently using `Pretty`

## 2. Medium: evaluator explain-mode visibility depends on `$` in names

Severity: Medium

Status:

- unresolved
- intentional convention
- semantically relevant for explain/debug output

Relevant code:

- [eval.go](/Users/cwd-k2/Projects/gicel/internal/eval/eval.go#L12)
- [eval.go](/Users/cwd-k2/Projects/gicel/internal/eval/eval.go#L18)
- [eval.go](/Users/cwd-k2/Projects/gicel/internal/eval/eval.go#L168)
- [eval.go](/Users/cwd-k2/Projects/gicel/internal/eval/eval.go#L235)
- [explain.go](/Users/cwd-k2/Projects/gicel/internal/eval/explain.go#L294)

Problem:

Compiler-generated names are detected by:

- `strings.Contains(name, "$")`

That predicate then controls whether bindings and patterns are treated as
user-visible in explanation output.

Why it matters:

- this is semantic classification by string convention
- if naming conventions expand or change, explain output silently changes with them
- the mechanism is implicit: no AST/Core flag says "generated by compiler"

Current mitigating factor:

- user identifiers do not appear to allow `$`, so collision with source names is currently avoided

Assessment:

This is not as severe as the `Pretty` cache-key fallback, because it only affects
observability rather than type checking.
Still, it is a real heuristic hack: generatedness is stored in the surface spelling.

Recommended direction:

- carry an explicit generated/user-visible flag in Core binders or observer metadata
- if not, at least centralize and document the naming contract

## 3. Medium: dictionary identity relies on lossy string mangling

Severity: Medium

Status:

- unresolved
- structurally risky
- affects instance naming, overlap diagnostics, and resolution plumbing

Relevant code:

- [class.go](/Users/cwd-k2/Projects/gicel/internal/check/class.go#L34)
- [class.go](/Users/cwd-k2/Projects/gicel/internal/check/class.go#L37)
- [class.go](/Users/cwd-k2/Projects/gicel/internal/check/class.go#L40)
- [instance_body.go](/Users/cwd-k2/Projects/gicel/internal/check/instance_body.go#L108)
- [instance_body.go](/Users/cwd-k2/Projects/gicel/internal/check/instance_body.go#L122)
- [instance_body.go](/Users/cwd-k2/Projects/gicel/internal/check/instance_body.go#L149)
- [instance.go](/Users/cwd-k2/Projects/gicel/internal/check/instance.go#L76)
- [resolve.go](/Users/cwd-k2/Projects/gicel/internal/check/resolve.go#L134)

Problem:

Dictionary naming and detection are string-encoded:

- `dictName(class)` = `class + "$Dict"`
- `classFromDict(name)` = `TrimSuffix(name, "$Dict")`
- instance dictionary bindings are named by `instanceDictName(className, typeArgs)`

The more brittle part is `typeNameForDict`.
It intentionally omits some information:

- type variables are dropped
- unknown/default cases collapse to `"?"`
- evidence rows are encoded only by a reduced string form via `evidenceRowName`

Why it matters:

- string names are doing identity work that should ideally belong to structured data
- omission means collisions are possible in principle
- future type extensions increase the chance that two distinct instance heads map to the same binding name

Example risk:

`evidenceRowName` records field count and labels or class names, but not full
field types, multiplicities, or row tails.
That is sufficient for some diagnostics, but it is not a collision-resistant identity.

Assessment:

This is not currently proven to misresolve instances, because overlap checks still
use structural overlap logic.
But it is an unnecessary fragility point and a clear heuristic encoding.

Recommended direction:

- separate human-readable dictionary names from internal unique identity
- keep a stable internal instance ID and generate display names from it if needed

## 4. Medium: tuple detection is encoded by record shape and label names

Severity: Medium

Status:

- intentional
- partly language encoding, partly heuristic rendering

Relevant code:

- [parse_expr.go](/Users/cwd-k2/Projects/gicel/internal/syntax/parse/parse_expr.go#L171)
- [parse_type.go](/Users/cwd-k2/Projects/gicel/internal/syntax/parse/parse_type.go#L280)
- [explain.go](/Users/cwd-k2/Projects/gicel/internal/eval/explain.go#L198)
- [explain.go](/Users/cwd-k2/Projects/gicel/internal/eval/explain.go#L297)

Problem:

Tuple-ness is inferred from record labels `_1`, `_2`, ...

This shows up in multiple places:

- parser-side constraint tuple desugaring
- runtime pretty-printing of tuple values
- pattern formatting for explain output

Why it matters:

- this is a shape heuristic, not a first-class tuple representation
- multiple subsystems must agree on the `_1`, `_2`, ... convention
- if tuple encoding changes, all of these consumers can drift separately

Assessment:

This is a weaker issue than the previous findings.
It is mostly an intentional encoding choice, not a hidden bug.
Still, it belongs in this review because it is exactly the sort of convention-based
coupling that grows accidental edge cases over time.

Recommended direction:

- if tuples remain encoded as records, centralize tuple-shape helpers and use them everywhere
- avoid duplicating `_1`/`_2` logic independently across parser, checker, and evaluator

## 5. Low: some parsing and explain helpers infer semantics from textual/shape conventions, but these are mostly acceptable sugar

Severity: Low

Status:

- intentional
- mostly acceptable
- worth tracking so they do not get mistaken for structural guarantees

Relevant code:

- [parse_type.go](/Users/cwd-k2/Projects/gicel/internal/syntax/parse/parse_type.go#L280)

- [matrix.go](/Users/cwd-k2/Projects/gicel/internal/check/exhaust/matrix.go#L252)

Problem:

There are helper paths that use best-effort shape recovery or textual sugar forms:

- `desugarConstraintTuple` recognizes tuple-like `Record` structure in types
- exhaustiveness witness reconstruction is explicitly best-effort
- witness formatting sorts record fields for stable rendering

Why it matters:

- these are not necessarily bugs
- but if future code starts reusing these helpers as semantic truth rather than UI/error support, they become the next generation of hack

Assessment:

These are currently acceptable because they are confined to parsing sugar,
error reporting, or witness rendering.
They should not be promoted into identity, equality, or solver logic.

## What I Did Not Find

The following stronger failure modes were not found in implementation code:

- direct equality checks of the form `types.Pretty(a) == types.Pretty(b)`
- semantic dispatch based on `Value.String()`
- cache keys built directly from `PrettyValue(...)`
- using `core.Pretty()` as a correctness or identity oracle

That matters because it means the main problem is not widespread carelessness.
It is a smaller set of specific convention-heavy hotspots.

## Test Strategy

The test plan should distinguish real semantic misuse from acceptable display logic.

### A. Structural-key tests

These tests should defend against any reintroduction of `Pretty` into identity paths.

Targets:

- `familyAppKey`
- any future structural caches over `types.Type`

Recommended tests:

- enumerate all current `types.Type` variants and assert `writeTypeKey` never falls through to display fallback
- add a regression test for keys involving `TyEvidence` and `TyError`
- if fallback remains temporarily, test and document it explicitly rather than leaving it silent

### B. Naming-convention boundary tests

These tests should make name-based heuristics explicit.

Targets:

- `$`-based generated-name filtering
- `$Dict` dictionary detection and round-tripping

Recommended tests:

- assert current lexer rules reject `$` in user identifiers
- add tests that explain-mode suppresses generated names and preserves source names
- add round-trip tests for `dictName` / `classFromDict` / `isDictName`

Important note:

These tests do not make the design ideal.
They just keep the current convention visible until explicit metadata replaces it.

### C. Collision-risk tests for dictionary naming

These tests should guard against accidental aliasing in string-mangled instance names.

Targets:

- `instanceDictName`
- `typeNameForDict`
- `evidenceRowName`

Recommended tests:

- build distinct instance heads that differ in row type, multiplicity, or tail and assert their names differ when the implementation intends them to
- add tests for unknown/default node kinds so they do not silently collapse to `"?"`
- if collisions are tolerated, document where structured overlap checks remain authoritative

### D. Convention-centralization tests

These tests should keep tuple-shape logic synchronized across subsystems.

Targets:

- tuple value rendering
- tuple pattern rendering
- tuple-like constraint desugaring

Recommended tests:

- use the same `_1`, `_2`, ... cases across parser, evaluator, and explain output
- add one negative test showing that non-sequential labels are not treated as tuples

## Recommended Near-Term Actions

1. Remove the `types.Pretty` fallback from [reduce.go](/Users/cwd-k2/Projects/gicel/internal/check/family/reduce.go#L327) and make structural key generation total.
2. Introduce explicit metadata for compiler-generated names in explain-mode paths.
3. Separate display names for dictionaries from internal identity used by instance plumbing.
4. Centralize tuple-shape detection into shared helpers instead of repeating `_1`, `_2`, ... logic.
5. Add tests that fail if display-oriented helpers start leaking into cache keys or identity logic again.

## Bottom Line

The codebase is not broadly comparing pretty-printed strings for equality.
That specific failure mode is mostly absent.

The real issue is narrower and more subtle:

- one confirmed `Pretty` fallback exists in structural cache key generation
- several internal mechanisms rely on naming conventions and lossy textual encodings
- some shape-based sugar encodings are acceptable today, but would become dangerous if reused outside parsing or presentation

That is fixable.
The repository does not need a philosophical rewrite; it needs a few targeted
replacements of implicit textual identity with explicit structured metadata.
