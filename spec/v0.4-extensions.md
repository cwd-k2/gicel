# Specification Draft v0.4 — Type System Extensions

This draft extends v0.3 with six features. All extensions elaborate to the existing 14-former Core IR; no new Core nodes are introduced.

**New keywords**: `class`, `instance`, `import` (total: 11)

**Design invariant**: Stratum 0 (pure/bind as computation algebra) is undisturbed. All extensions are refinements or additions per the v0.3 extension framework (§4).

---

## 1. Kind System Extensions

### 1.1 Extended Kind Grammar

```
Kind ::= 'Type'
       | 'Row'
       | 'Constraint'          -- NEW: §2
       | Kind '->' Kind
       | UserKind              -- NEW: §4 (DataKinds)

UserKind ::= <name of a promoted data type>
```

`Constraint` is a first-class kind. Values of kind `Constraint` are type class predicates (e.g., `Eq Bool : Constraint`).

### 1.2 Kind Checking

A type class `C` with `n` parameters has kind `T1 -> ... -> Tn -> Constraint`. Application `C a1 ... an` has kind `Constraint` when each `ai` has the kind expected by `C`.

## 2. Type Classes

### 2.1 Vocabulary Position

Activates the reserved **Constraint** vocabulary item (v0.3 §3.6, Stratum 3). This is an **Addition** in the extension framework.

### 2.2 Surface Syntax

#### Class Declaration

```
class <Constraint>* <Name> <tyvar>+ {
  <method> :: <Type> ;
  ...
}
```

Superclass constraints precede the class name:

```
class Eq a => Ord a {
  compare :: a -> a -> Ordering
}
```

Multiple superclasses use curried form:

```
class C1 a => C2 a => MyClass a {
  ...
}
```

#### Instance Declaration

```
instance <Constraint>* <Name> <Type>+ {
  <method> := <Expr> ;
  ...
}
```

Instance constraints (context):

```
instance Eq a => Eq (Maybe a) {
  eq := \x -> \y -> case x {
    Nothing -> case y { Nothing -> True; Just _ -> False };
    Just a  -> case y { Nothing -> False; Just b -> eq a b }
  }
}
```

#### Qualified Types

The `=>` token introduces constraints in type expressions. It is right-associative and binds looser than `->` but tighter than `forall ... .`:

```
Type ::= 'forall' TyBinders '.' Type
       | Constraint '=>' Type            -- NEW
       | Type '->' Type
       | ...

Constraint ::= TypeApp                   -- e.g., Eq a, Coercible a b
```

Example:

```
f :: forall a. Eq a => a -> a -> Bool
```

Multiple constraints, curried:

```
g :: forall a b. Eq a => Ord b => a -> b -> Bool
```

### 2.3 Multi-Parameter Type Classes

Supported from the start (required for `Coercible`):

```
class Coercible a b {
  coerce :: a -> b
}
```

Kind of `Coercible`: `Type -> Type -> Constraint`.

### 2.4 Elaboration

Type classes elaborate entirely to existing Core IR constructs via **dictionary passing**. No new Core nodes are needed.

#### Class → Data Type + Selectors

```
class Eq a {
  eq :: a -> a -> Bool
}
```

Elaborates to:

```
data Eq$Dict a = Eq$MkDict (a -> a -> Bool)

eq :: forall a. Eq$Dict a -> a -> a -> Bool
eq := \d -> case d { Eq$MkDict f -> f }
```

General rule: a class with `n` methods and `m` superclasses produces a data type with one constructor of arity `m + n`. The first `m` fields are superclass dictionaries.

```
class Eq a => Ord a {
  compare :: a -> a -> Ordering
}
```

Elaborates to:

```
data Ord$Dict a = Ord$MkDict (Eq$Dict a) (a -> a -> Ordering)

-- Superclass selector
ordEq :: forall a. Ord$Dict a -> Eq$Dict a
ordEq := \d -> case d { Ord$MkDict e _ -> e }

-- Method selector
compare :: forall a. Ord$Dict a -> a -> a -> Ordering
compare := \d -> case d { Ord$MkDict _ f -> f }
```

#### Instance → Dictionary Value

```
instance Eq Bool {
  eq := \x -> \y -> case x {
    True  -> y;
    False -> case y { True -> False; False -> True }
  }
}
```

Elaborates to:

```
eq$Bool :: Eq$Dict Bool
eq$Bool := Eq$MkDict (\x -> \y -> case x {
  True  -> y;
  False -> case y { True -> False; False -> True }
})
```

#### Constrained Instance → Dictionary Function

```
instance Eq a => Eq (Maybe a) { ... }
```

Elaborates to:

```
eq$Maybe :: forall a. Eq$Dict a -> Eq$Dict (Maybe a)
eq$Maybe := \da -> Eq$MkDict (\x -> \y -> ...)
```

#### Call Site → Dictionary Insertion

```
eq True False
```

The checker resolves `Eq Bool` and inserts the dictionary:

```
(eq @Bool) eq$Bool True False
```

### 2.5 Instance Resolution

**Algorithm**: For a goal `C T1 ... Tn`:

1. Search all in-scope instances for `C`.
2. For each instance `instance Ctx => C S1 ... Sn`, attempt to unify `Ti ~ Si` for all `i`.
3. If exactly one instance matches, use it. Recursively resolve any context constraints.
4. If zero instances match: error (no instance).
5. If multiple instances match: error (overlapping instances).

**No backtracking**: resolution is greedy. No overlapping instances are permitted.

**Orphan instances**: allowed (embedded language with controlled namespace).

**No default methods**: every method must be defined in every instance.

### 2.6 Interaction with Computation Types

Constraints are value-level functions (dictionary arguments). They compose freely with `Computation`:

```
f :: forall a. Eq a => a -> Computation {} {} Bool
-- elaborates to:
f :: forall a. Eq$Dict a -> a -> Computation {} {} Bool
```

Constraints do not affect the `pre`/`post` row structure.

## 3. Coercible

### 3.1 Definition

`Coercible` is a multi-parameter type class with manual instances:

```
class Coercible a b {
  coerce :: a -> b
}
```

No automatic instance derivation (no newtype in the language). Instances are written explicitly by the user or provided as assumptions from the host.

### 3.2 Example

```
instance Coercible (Pair a b) (Pair b a) {
  coerce := \p -> case p { Pair x y -> Pair y x }
}
```

## 4. DataKinds

### 4.1 Vocabulary Position

**Refinement** of Kind (v0.3 §3.2.3): `Kind` gains user-declared members from promoted data constructors.

### 4.2 Promotion Rules

Every `data` declaration automatically promotes its constructors to the type level:

```
data DBState = Opened | Closed
```

Produces:

| Level | Name | Classification |
|-------|------|---------------|
| Term | `Opened` | Constructor, type `DBState` |
| Term | `Closed` | Constructor, type `DBState` |
| Type | `Opened` | Type, kind `DBState` |
| Type | `Closed` | Type, kind `DBState` |
| Kind | `DBState` | Kind (with inhabitants `Opened`, `Closed`) |

### 4.3 Namespace Resolution (Same-Namespace, Option B)

In type position, a name is resolved by:

1. Check registered type constructors and type aliases.
2. Check promoted data constructors.
3. If both match, emit an ambiguity error.

Since the language has a controlled namespace (embedded, no large prelude), ambiguity is rare. If it arises, the user must rename.

### 4.4 Kind-Indexed Types

With DataKinds, types can be parameterized by promoted kinds:

```
data DBState = Opened | Closed

-- Host-registered: DB : DBState -> Type
-- Then:
--   DB Opened : Type
--   DB Closed : Type
```

This enables precise capability tracking in rows:

```
dbOpen :: forall r. Computation { db : DB Closed | r }
                                { db : DB Opened | r }
                                Unit
```

### 4.5 Kind Checking for Promoted Constructors

When a data type `D` with parameters `p1 : K1, ..., pn : Kn` is declared, each constructor `C` with fields `F1, ..., Fm` (all of kind `Type`) produces a promoted type `C` of kind `K1 -> ... -> Kn -> D` only when `m = 0` (nullary constructors).

For constructors with fields, promotion is not applicable (term-level data, not type-level).

**Rationale**: Promoting constructors with fields would require type-level computation (type families), which is a phase transition. DataKinds is strictly a refinement.

### 4.6 Interaction with Existing Kind System

```
Kind ::= Type | Row | Constraint | Kind -> Kind | UserKind

-- UserKind examples after promotion:
-- DBState, Bool, Ordering, ...
```

The kind checker extends `Equal` on kinds to handle `UserKind` by name equality.

## 5. GADTs

### 5.1 Vocabulary Position

**Refinement** of Algebraic Data (v0.3 §3.3.1) and Equality (v0.3 §3.4.1): constructors may have refined return types; case elimination introduces local type equalities.

### 5.2 Surface Syntax

GADTs use `= {` to distinguish from regular ADTs:

```
-- Regular ADT (unchanged)
data Maybe a = Just a | Nothing

-- GADT
data Expr a = {
  BoolLit :: Bool -> Expr Bool;
  IntLit  :: Int -> Expr Int;
  If      :: Expr Bool -> Expr a -> Expr a -> Expr a
}
```

Grammar:

```
DeclData ::= 'data' ConName TyBinder* '=' ConDecl ('|' ConDecl)*    -- ADT
           | 'data' ConName TyBinder* '=' '{' GADTCon (';' GADTCon)* '}'  -- GADT

GADTCon ::= ConName '::' Type
```

Each GADT constructor's type must return the declared data type (possibly with refined indices).

### 5.3 Type Equality Refinement

When pattern matching on a GADT constructor, the checker introduces local type equalities derived from unifying the scrutinee type with the constructor's return type.

```
eval :: Expr Bool -> Bool
eval := \e -> case e {
  BoolLit b -> b;          -- a ~ Bool known, so b : Bool, return Bool OK
  If c t f  -> eval (If c t f)
}
```

**Mechanism**: for each branch with pattern `C p1 ... pn`:

1. Instantiate `C`'s declared type with fresh metavariables.
2. Unify the constructor's return type with the scrutinee type.
3. The resulting substitution provides local type equalities.
4. Check the branch body under these equalities.

### 5.4 Exhaustiveness with GADTs

A constructor is **relevant** for a scrutinee type `T` if its return type can unify with `T`. Irrelevant constructors need not be covered:

```
eval :: Expr Bool -> Bool
eval := \e -> case e {
  BoolLit b -> b;
  If c t f  -> ...
}
-- IntLit is irrelevant: Expr Int does not unify with Expr Bool
```

The exhaustiveness checker attempts unification of each constructor's return type with the scrutinee type. Only relevant constructors are required.

### 5.5 Elaboration

GADTs elaborate to the same Core IR as regular ADTs. The refined typing is enforced during checking and erased at runtime. No coercion terms are needed because the evaluator interprets Core IR directly (no compilation to a lower-level representation).

Core `DataDecl` is extended with optional constructor return types for kind/type tracking during elaboration, but at runtime, pattern matching is identical to regular ADTs.

### 5.6 Restrictions

- No nested GADTs in v0.4 (constructor fields must be regular types).
- No existential quantification in constructors (defer to later draft).

## 6. Module System

### 6.1 Design Scope

Minimal module system for an embedded language. Modules are named source bundles registered with the engine. `import` makes a module's definitions available.

### 6.2 Host API

```go
// Register a module's source code with the engine.
// The module is parsed and type-checked at registration time.
eng.RegisterModule("MyLib", source)
```

### 6.3 Surface Syntax

```
import ModuleName
```

`import` must appear at the top of a source file, before any other declarations.

Grammar:

```
Program  ::= Import* Decl*
Import   ::= 'import' ModuleName
ModuleName ::= ConName
```

### 6.4 Semantics

- `import M` brings all top-level definitions from module `M` into scope.
- Modules are compiled independently. A module's type-checked bindings (types, constructors, class instances, type aliases) are available to importers.
- Circular imports are forbidden (topological sort at registration time).
- No qualified names in v0.4. All imports are unqualified.
- Name collisions between imported modules are an error at the import site.
- The prelude is implicitly imported (as in v0.3). Explicit `import Prelude` is a no-op.

### 6.5 Module Compilation

When `eng.RegisterModule(name, source)` is called:

1. Parse the source.
2. Type-check against the engine's registered types, assumptions, and previously registered modules.
3. Store the module's exports: data types, constructors, type aliases, value bindings (with types), class declarations, instance declarations.

When a user source imports the module, these exports are injected into the checker's context before processing the user's declarations.

### 6.6 No Export Control

All top-level definitions in a module are exported. Selective exports are deferred to a future draft.

## 7. Standard Library

### 7.1 Prelude (unchanged from v0.3 + Effect alias)

```
data Bool = True | False
data Unit = Unit
data Result e a = Ok a | Err e
data Pair a b = Pair a b
data Maybe a = Just a | Nothing
data List a = Cons a (List a) | Nil
type Effect r a = Computation r r a
```

### 7.2 Stdlib (requires type classes)

Defined as a module `Std` (or inlined into the prelude):

```
data Ordering = LT | EQ | GT

class Eq a {
  eq :: a -> a -> Bool
}

class Eq a => Ord a {
  compare :: a -> a -> Ordering
}

class Functor f {
  fmap :: forall a b. (a -> b) -> f a -> f b
}

class Foldable t {
  foldr :: forall a b. (a -> b -> b) -> b -> t a -> b
}
```

### 7.3 Prelude Instances

```
instance Eq Bool {
  eq := \x -> \y -> case x {
    True  -> y;
    False -> case y { True -> False; False -> True }
  }
}

instance Eq Unit {
  eq := \_ -> \_ -> True
}

instance Eq Ordering {
  eq := \x -> \y -> case x {
    LT -> case y { LT -> True;  EQ -> False; GT -> False };
    EQ -> case y { LT -> False; EQ -> True;  GT -> False };
    GT -> case y { LT -> False; EQ -> False; GT -> True  }
  }
}

instance Eq a => Eq (Maybe a) {
  eq := \x -> \y -> case x {
    Nothing -> case y { Nothing -> True; Just _  -> False };
    Just a  -> case y { Nothing -> False; Just b -> eq a b }
  }
}

instance Eq a => Eq b => Eq (Pair a b) {
  eq := \x -> \y -> case x {
    Pair a1 b1 -> case y {
      Pair a2 b2 -> case eq a1 a2 {
        True  -> eq b1 b2;
        False -> False
      }
    }
  }
}

instance Functor Maybe {
  fmap := \f -> \x -> case x {
    Nothing -> Nothing;
    Just a  -> Just (f a)
  }
}

instance Functor List {
  fmap := \f -> \xs -> case xs {
    Nil       -> Nil;
    Cons x rest -> Cons (f x) (fmap @List f rest)
  }
}

instance Foldable Maybe {
  foldr := \f -> \z -> \x -> case x {
    Nothing -> z;
    Just a  -> f a z
  }
}

instance Foldable List {
  foldr := \f -> \z -> \xs -> case xs {
    Nil       -> z;
    Cons x rest -> f x (foldr @List f z rest)
  }
}
```

**Note on recursion**: `fmap` and `foldr` for `List` are recursive. These instances require `EnableRecursion()` or must be provided as host-side assumptions.

### 7.4 Generalized fold

`Foldable` subsumes the v0.3 `fold` design (host-provided per data type). With type classes:

- Each data type can have a `Foldable` instance.
- `foldr` is the canonical fold operation.
- Instances may be user-defined (with `rec`) or host-provided (as assumptions).

## 8. Grammar Summary

### 8.1 New Tokens

| Token | Use |
|-------|-----|
| `class` | Keyword: class declaration |
| `instance` | Keyword: instance declaration |
| `import` | Keyword: module import |
| `=>` | Constraint arrow (in types) |

### 8.2 Extended Grammar (additions to v0.3)

```
Program  ::= Import* Decl*

Import   ::= 'import' ConName

Decl     ::= ...                                           -- existing
           | DeclClass                                     -- NEW
           | DeclInstance                                   -- NEW

DeclClass ::= 'class' Constraint* ConName TyVar+ '{' ClassMethod (';' ClassMethod)* '}'
ClassMethod ::= VarName '::' Type

DeclInstance ::= 'instance' Constraint* ConName Type+ '{' InstMethod (';' InstMethod)* '}'
InstMethod ::= VarName ':=' Expr

DeclData ::= 'data' ConName TyBinder* '=' ConDecl ('|' ConDecl)*        -- ADT (unchanged)
           | 'data' ConName TyBinder* '=' '{' GADTCon (';' GADTCon)* '}'  -- GADT (NEW)

GADTCon ::= ConName '::' Type

Type     ::= 'forall' TyBinders '.' Type
           | Constraint '=>' Type                          -- NEW
           | Type '->' Type
           | TypeApp
           | ...

Constraint ::= TypeApp

Kind     ::= 'Type' | 'Row' | 'Constraint' | Kind '->' Kind | ConName
```

### 8.3 Precedence of Type Operators

From loosest to tightest:

1. `forall ... .` (binds body to the right)
2. `=>` (right-associative)
3. `->` (right-associative)
4. Type application (left-associative)

## 9. Core IR Impact

**No new Core nodes.** All six features elaborate to the existing 14 formers:

| Feature | Elaboration Target |
|---------|--------------------|
| Type class declaration | `DataDecl` + `Binding` (selectors) |
| Instance declaration | `Binding` (dictionary value/function) |
| Constrained type | Implicit `Lam`/`App` (dictionary passing) |
| DataKinds | Kind-level only; no Core representation |
| GADT | Same `DataDecl` + `Case`; typing erased at runtime |
| Module import | Declarations spliced into checker context |
| Coercible | Regular type class (same as above) |

## 10. Implementation Phases

### Phase 1: Constraint Kind + Type Classes

- Extend kind grammar with `Constraint`
- Parse `class` and `instance` declarations
- Parse `=>` in type expressions
- Elaborate classes to dictionary data types + selectors
- Elaborate instances to dictionary values
- Instance resolution in the checker
- Dictionary insertion at call sites

### Phase 2: DataKinds

- Promote data constructors to type level
- Extend kind checking for user-declared kinds
- Same-namespace resolution in type position

### Phase 3: GADTs

- Parse GADT syntax `data T a = { ... }`
- Type equality refinement in case branches
- Updated exhaustiveness checking for GADT relevance

### Phase 4: Standard Library + Foldable

- Define `Eq`, `Ord`, `Functor`, `Foldable` classes
- Write instances for prelude types
- Integrate as module or extended prelude

### Phase 5: Module System

- `RegisterModule` on Engine
- Parse `import` declarations
- Module compilation and export storage
- Import resolution in checker

### Phase 6: Coercible

- Define `Coercible` as multi-param type class
- Manual instance definitions

## 11. Interaction Matrix

| Feature | Rows | Computation | forall | case |
|---------|------|-------------|--------|------|
| Type classes | Constraint in row? No — Constraints are not row entries | Dict args are values, no effect on pre/post | Constraints follow forall | No interaction |
| DataKinds | Row field types may use promoted kinds | Promoted types in pre/post rows | Promoted kinds in binders | No interaction |
| GADTs | No direct interaction | No direct interaction | GADT constructors may be polymorphic | **Key interaction**: case introduces local equalities |
| Modules | No interaction | No interaction | No interaction | No interaction |

## 12. Open Questions (to resolve during implementation)

1. **Instance search scope**: Are instances from imported modules automatically in scope? (Proposed: yes, like Haskell.)
2. **Recursive instances**: `instance Eq a => Eq (List a)` requires `rec`. Should instances implicitly enable recursion for their methods?
3. **Kind inference for promoted types**: When `data S = A | B` is promoted, is `S` inferred as a kind automatically, or does it require explicit annotation in type signatures?
4. **GADT + row interaction**: Can GADT constructors refine row types? (Proposed: defer to later draft.)
