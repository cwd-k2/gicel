# Specification Draft v0.7 — Records, Tuples, HKT, Evidence Sort

This draft extends v0.5 with four features. Records and tuples elaborate to PrimOp nodes; HKT extends the kind system; Evidence Sort is an internal unification of row representations.

**New keywords**: none (total: 11)

**New tokens**: `!#` (record projection)

**Removed types**: `Unit`, `Pair`

---

## 1. Records

### 1.1 Vocabulary Position

**Addition** of a product type based on the existing Row kind infrastructure.

### 1.2 Type

`Record` is a built-in type constructor of kind `Row → Type`. Record types are written by applying `Record` to a row:

```
Record { x : Int, y : Bool }
Record { x : Int | r }
Record {}
```

### 1.3 Literals

Record literals use `=` to bind fields, separated by `,`:

```
{ x = 1, y = True }
{ x = 1, y = True, z = "hello" }
{}
```

A record literal `{ l₁ = e₁, ..., lₙ = eₙ }` has type `Record { l₁ : T₁, ..., lₙ : Tₙ }` where each `eᵢ : Tᵢ`.

### 1.4 Projection

The `!#` operator projects a field from a record:

```
r!#x            -- project field x
r!#x!#y         -- chained projection (left-associative, atom-level precedence)
f r!#x          -- f (r!#x) — projection binds tighter than application
```

Typing rule: if `e : Record { l : T | r }`, then `e!#l : T`.

### 1.5 Update

Record update uses `|` inside braces:

```
{ r | x = 42 }
{ r | x = 42, y = True }
```

Typing rule: if `e : Record { x : T₁ | r }`, then `{ e | x = e' } : Record { x : T₂ | r }` where `e' : T₂`. The field must exist in the original record.

### 1.6 Pattern Matching

Record patterns are **open** (partial match permitted):

```
\{ x = a, y = b } -> a
\{ x = a } -> a               -- other fields ignored
case r { { x = a, y = b } -> a }
{ x = n } := r                -- block binding destructuring
```

Patterns may nest:

```
\{ pos = (x, y), name = n } -> x
```

### 1.7 Disambiguation

In expression position, `{` followed by:
- `ident :=` → block expression
- `ident =` → record literal
- `expr |` → record update

In type position, `{` always begins a row type.

### 1.8 Elaboration

Records elaborate to PrimOp nodes. No new Core IR formers.

| Operation | PrimOp |
|-----------|--------|
| Literal | `RecordCon` |
| Projection | `RecordProj` |
| Update | `RecordUpdate` |

Runtime representation: `map[string]Value`.

### 1.9 Field Order

Field order is semantically irrelevant (Row property). `{ x = 1, y = 2 }` and `{ y = 2, x = 1 }` are equal.

## 2. Tuples

### 2.1 Definition

Tuples are syntactic sugar for records with positional labels `_0`, `_1`, `_2`, ....

### 2.2 Desugaring

| Surface | Desugars to |
|---------|-------------|
| `(1, True)` | `{ _0 = 1, _1 = True }` |
| `(Int, Bool)` | `Record { _0 : Int, _1 : Bool }` |
| `t!#_0` | record projection on `_0` |
| `(a, b)` (pattern) | `{ _0 = a, _1 = b }` (pattern) |

### 2.3 Syntax

In expression position, `(` followed by an expression and then `,` indicates a tuple:

```
(1, True)                -- 2-tuple
(1, True, "hello")       -- 3-tuple
(1)                      -- grouping (not a 1-tuple)
```

In type position, `(` with `,` indicates a tuple type:

```
(Int, Bool)              -- 2-tuple type
(Int, Bool, String)      -- 3-tuple type
(Int)                    -- grouping
```

### 2.4 Unit Replacement

`()` is the 0-tuple, equivalent to the empty record `{}`. It replaces the former `Unit` type and constructor. No alias is provided.

The type `()` has kind `Type` (sugar for `Record {}`). The value `()` is the empty record.

### 2.5 Pair Replacement

`Pair a b` is removed. The 2-tuple `(a, b)` replaces it.

```
fst := \(a, _) -> a
snd := \(_, b) -> b
```

## 3. Higher-Kinded Types

### 3.1 Kind Variables

Kind variables are introduced with explicit annotation in `forall` binders:

```
forall (k : Kind). forall (f : k -> Type). ...
```

`Kind` is a distinguished sort. Kind variables range over kinds (`Type`, `Row`, `Constraint`, `K₁ → K₂`, user kinds).

### 3.2 Kind Unification

Kind inference is extended with kind metavariables and unification:

- `KindMeta` metavariables in the ordered context
- Kind occurs check
- Kind substitution (`applyKindSubst`)

Kind variables declared in `forall` are explicit and never inferred.

### 3.3 Poly-Kinded Type Classes

Type class parameters may be kind-polymorphic:

```
class Functor (f : k -> Type) {
  fmap :: forall a b. (a -> b) -> f a -> f b
}
```

Instance resolution matches kind arguments structurally.

### 3.4 Interaction with Existing Features

- `forall` binders may mix type and kind parameters: `forall (k : Kind) (f : k -> Type) a. ...`
- DataKinds promoted types participate in kind unification
- Kind equality uses unification rather than structural comparison when kind variables are present

### 3.5 What Does NOT Change

- Kind inference for programs without kind variables remains unchanged
- No kind polymorphism inference — kind variables must be explicitly bound
- Type families are not introduced (door remains open)

## 4. Evidence Sort (Internal)

### 4.1 Overview

Unify `TyRow` and `TyConstraintRow` into `TyEvidenceRow` parameterized by an extensible `EvidenceFiber` interface.

### 4.2 Structure

```
TyEvidenceRow {
    Fiber   EvidenceFiber
    Fields  []EvidenceField
    Tail    Type
}

EvidenceFiber interface {
    MatchLabels(a, b) bool
    UnifyEntries(a, b) error
    KindTag() string
}
```

Initial fibers: `CapabilityFiber`, `ConstraintFiber`.

### 4.3 Invariant

All existing programs continue to type-check identically. This is a refactoring with no user-visible changes.

## 5. Module / Host Boundary

### 5.1 Prelude Split

The current Prelude is split into:
- **Core** (not replaceable): language-essential definitions
- **Prelude** (replaceable): standard library types, classes, instances

### 5.2 New Engine API

```go
eng.SetPrelude(source string)         // Replace default Prelude
eng.RegisterModuleFile(path string)   // Load module from file
```

### 5.3 Package Promotion

Selected internal packages are promoted to public API surface.

## 6. Prelude Changes

### 6.1 Removed

```
data Unit = Unit
data Pair a b = Pair a b
```

### 6.2 Added / Changed

```
-- () is built-in (0-tuple = empty record)
-- (a, b) replaces Pair a b

fst :: forall a b. (a, b) -> a
fst := \(a, _) -> a

snd :: forall a b. (a, b) -> b
snd := \(_, b) -> b
```

Instances formerly on `Unit` migrate to `()`. Instances formerly on `Pair a b` migrate to `(a, b)`.

## 7. Grammar Changes

### 7.1 New Token

| Token | Meaning |
|-------|---------|
| `!#` | Record projection |

### 7.2 Extended Grammar

```
Expr     ::= ...
           | '{' FieldBind (',' FieldBind)* '}'       -- record literal
           | '{' Expr '|' FieldBind (',' FieldBind)* '}'  -- record update
           | Expr '!#' LowerName                       -- projection
           | '(' Expr ',' Expr (',' Expr)* ')'        -- tuple literal
           | '(' ')'                                   -- unit literal

FieldBind ::= LowerName '=' Expr

Pattern  ::= ...
           | '{' FieldPat (',' FieldPat)* '}'         -- record pattern
           | '(' Pattern ',' Pattern (',' Pattern)* ')' -- tuple pattern
           | '(' ')'                                   -- unit pattern

FieldPat ::= LowerName '=' Pattern

Type     ::= ...
           | '(' Type ',' Type (',' Type)* ')'        -- tuple type

Kind     ::= 'Type' | 'Row' | 'Constraint' | 'Kind'  -- Kind sort (NEW)
           | Kind '->' Kind
           | ConName                                   -- promoted DataKinds
           | KindVar                                   -- kind variable (NEW)
```

### 7.3 Precedence

`!#` binds at atom level (tighter than function application).

---

## 8. Core IR Impact

**No new Core nodes.** Records elaborate to PrimOp. HKT extends the kind system only. Evidence Sort is internal refactoring.
